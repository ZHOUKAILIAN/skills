- **个人/基础设施项目** 包括 `agent-team-runtime`、`ai-team-runtime`、`skills` 和其他 `mySelf` 仓库。事实来源应是该仓库自己的 README、测试、运行时产物、状态文件和 issue。除非用户明确把个人项目和 CrewPals 连接起来，否则不要套用 CrewPals 专属的飞书、生产日志、SLS/MySQL 或产品文档要求。
- **CrewPals 工作项目** 包括 `crewpals-mp`、`group_pals` 和相关 worktree。任务匹配时，应用 CrewPals 的文档优先、飞书、环境、日志、数据和 E2E 验证习惯。

错误：修改 `agent-team-runtime` 前强制要求更新飞书 PRD。
正确：对 `agent-team-runtime` 使用该仓库自己的文档、测试和 trace 产物；只有在 CrewPals 工作项目内或用户明确要求时，才使用飞书/CrewPals 流程。

### 默认偏好

- 在提出架构或写代码前，先阅读仓库、文档、日志和既有模式。
- 优先使用工程控制，而不是依赖提示词记忆。如果 JSON/YAML/状态机校验可以强制行为，就使用它，而不是要求模型记住。
- 面向人的 PRD、技术方案、报告和飞书文档使用 Markdown。机器状态、契约、skill 注入、trace 数据和工作流控制使用 JSON/YAML。
- 避免重复控制文件。如果两个配置表达同一个事实，应先合并或清晰定义所有权，再改变行为。
- CrewPals 的 PRD、技术方案、调查报告和交接产物默认使用中文，除非仓库约定要求英文。个人或开源风格项目按该仓库已有语言和受众来写。
- 对流程或架构说明，使用流程图、PlantUML/Mermaid、表格和明确示例。陌生英文术语用中文解释。
- 除非用户要求或既有模板要求，不添加 `Non-Goals` 或 `非目标` 章节。
- 对 CrewPals 已有功能或回归问题，默认更新归属需求/设计文档，而不是新建孤立的 bugfix 文档。个人基础设施项目则更新仓库原生的设计记录、README、issue、changelog 或测试计划，不强行套 CrewPals 文档对。
- 对多阶段工作保留可追溯性：原始用户输入、阶段交接、选中的 skills 及其来源路径或 URL、实际发送给 agent 的提示词、产物、验证输出和最终决策。
- 实现工作中，创建或使用用户要求的分支/worktree，保持变更聚焦，不回退无关的用户改动。
- 后端/API 工作中，单元测试通常不够。可行时应启动服务或用代表性数据调用目标 endpoint。
- bug 调查要产出症状、根因、影响、证据和修复/下一步。不要停在貌似合理的解释。
- 对完成声明，报告验证命令和结果。如果跳过某项检查，说明原因。
